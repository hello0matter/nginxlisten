# Nginx Trace Hunter

面向应急响应和溯源排查的 Nginx 访问日志分析工具。默认适合放在软件根目录或日志目录下直接运行，用本地规则发现日志、登记状态、增量巡检，并在需要时输出风险报告和告警推送。

## 快速开始

最常用方式是不传参数：

```powershell
python nginx_trace_hunter.py
```

这会进入驻场增量巡检模式：

- 默认扫描当前目录。
- 自动发现 `*.access.log`、`access.log`、`*access*.log`。
- 自动登记已发现日志和读取位置。
- 后续只分析新增内容。
- 输出目录默认是当前目录下的 `hunter_output`。

只想跑一次历史分析：

```powershell
python nginx_trace_hunter.py --once
```

打开图形界面：

```powershell
python nginx_trace_hunter.py --gui
```

指定日志目录或单个日志文件：

```powershell
python nginx_trace_hunter.py D:\logs --once
python nginx_trace_hunter.py D:\logs\access.log --once
```

## GUI 使用

GUI 适合不想记参数时使用：

- `监控日志目录/文件`：列表框管理多个监控目标，可以是软件根目录、日志目录、单个日志文件或压缩包；旧配置里的英文分号 `;` 和每行一个路径都兼容。
- `打开日志`：直接打开输出目录里的 `hunter.log`。
- `打开输出`：直接打开输出目录。
- `添加目录`：追加一个目录到监控目标列表。
- `添加文件`：追加一个日志文件或压缩包到监控目标列表。
- `编辑目标`：替换当前选中的监控目标。
- `删除目标`：删除当前选中的监控目标。
- `上移 / 下移`：调整多个监控目标的顺序。
- `清空`：清空所有监控目标。
- `配置设置`：打开高级配置弹窗，设置输出目录、状态文件、日期范围、AI、提示词和告警 Webhook。

高级配置弹窗包含：

- `输出目录`：报告、CSV、运行日志、状态文件默认都会落到这里。
- `状态文件`：驻场模式记录已发现日志、读取偏移、上次告警等状态。
- `轮询秒数`：驻场模式每隔多少秒检查一次新增日志。
- `首次接管尾部MB`：首次遇到超大日志时，只从尾部指定大小开始接管，避免第一次扫描几十 GB。
- `重新发现日志轮数`：驻场运行若干轮后重新递归发现新日志。
- `起始日期 / 结束日期`：主要用于一次性历史分析，可用旁边的 `选日期` 按钮选择。
- `启用 AI 研判`：默认关闭；开启后只在本地规则发现明显风险时做辅助总结。
- `AI 提示词`：可直接编辑 AI 研判的系统提示词，用于控制输出风格、关注点和报告格式。
- `启用告警`：开启后可配置钉钉、企微、飞书 Webhook。
- `钉钉关键词`：钉钉机器人开启“自定义关键词”时填写，推送前会自动放在消息最前面。

GUI 按钮含义：

- `运行一次`：按当前路径和日期范围做一次全量历史分析，跑完退出。
- `启动驻场`：启动增量巡检，持续检查新增日志。
- `保存配置`：只保存当前 GUI 配置，不启动扫描。

GUI 中的配置会自动保存。编辑路径、日期、AI 提示词、Webhook、钉钉关键词、告警阈值等字段后，程序会在短暂延迟后写入配置文件；也可以点击 `保存配置` 立即保存。

## 全量分析与增量巡检

这两个概念容易混淆：

- `全量分析`：对当前选定路径和日期范围内的日志完整分析一遍。对应 `--once` 或 GUI 的 `运行一次`。
- `增量巡检`：驻场模式下只读取上次扫描位置之后新增的内容。对应默认启动或 GUI 的 `启动驻场`。

日志里出现的阶段含义：

- `全量统计阶段`：先读日志做 IP、URI、状态码、响应大小、时间窗口等统计。
- `异常复核阶段`：再抽取异常样本，例如异常大响应、高频接口、扫描探测、ID 枚举等。

这两个阶段不是“全量/增量”的意思，而是一次分析内部的两个处理阶段。

## 日期范围

日期范围用于历史数据很多时缩小扫描范围，特别适合目录里有大量按天压缩包或历史日志的场景。

支持来源：

- 从日志内容里的时间字段提取日期。
- 从文件名或路径中的 `YYYY-MM-DD`、`YYYYMMDD` 等日期线索过滤。

注意：

- `运行一次` 时日期范围最有用。
- `启动驻场` 主要关注新增内容，日期范围不是主要控制手段。

## 输出文件

每次成功分析后，输出目录通常包含：

- `report.md`：人工可读报告。
- `report.json`：完整结构化结果。
- `suspicious_ips.csv`：风险 IP 明细。
- `suspicious_uris.csv`：风险 URI 明细。
- `review_ips.csv`：待人工研判 IP 分层列表。
- `hunter.log`：运行日志，排查卡顿和异常时优先查看。
- `hunter_agent.json`：驻场状态文件，记录已登记日志、偏移、告警状态等。

脚本不会执行封禁动作，也不会生成防火墙、iptables、nginx deny 等自动封禁脚本。`review_ips.csv` 和推送里的 IP 只用于安全人员人工研判。

## 告警推送

支持三类 Webhook 文本推送：

- 钉钉
- 企业微信
- 飞书

推送内容会尽量具体到文件，包括：

- 扫描文件数。
- 监控目标汇总，例如每个日志目录/压缩包的请求量和文件数。
- 重点涉及文件。
- 高优先级研判 IP。
- 观察 IP。
- 高风险 URI。
- 热点时间窗口。
- 主要风险依据。

钉钉机器人如果启用了自定义关键词，需要在 GUI 的 `钉钉关键词` 里配置。发送钉钉前，脚本会自动把关键词放到正文第一行。

## AI 研判

AI 是辅助总结，不是基础扫描引擎：

- 不负责发现日志。
- 不负责判断哪些文件是日志。
- 不负责登记状态。
- 不会默认开启。
- 只有开启 AI 且本地规则判断风险达到阈值时才会调用。
- GUI 里的 `AI 提示词` 会持久化保存，后续重启软件仍会沿用。

这样设计是为了保证性能、稳定性和可审计性。基础能力全部由本地规则完成，AI 只用于帮助总结风险和给出研判建议。

## 日志发现逻辑

默认识别这些 access log：

- `*.access.log`
- `access.log`
- `*access*.log`

默认排除或降低优先级：

- `error.log`
- `*.error.log`
- 明显静态资源请求

脚本支持把软件根目录作为输入路径。它会递归查找日志文件、Nginx 配置、常见源码目录和归档文件，用于做“日志 URI -> 配置/应用目录”的反查。

## 归档文件

支持常见归档：

- `.zip`
- `.tar`
- `.tgz`
- `.tar.gz`
- `.gz`
- `.7z`
- `.rar`

说明：

- `zip`、`tar`、`gz` 主要依赖 Python 标准库。
- `7z`、`rar` 通常需要系统里能调用 `7z`。
- 归档会解到输出目录的临时提取目录中参与分析。

## Windows 与 Linux

脚本主路径使用 Python 标准库，Windows 和 Linux 都可以运行。

默认配置文件位置：

- Windows：`%APPDATA%\nginx_trace_hunter\config.json`
- Linux：`~/.config/nginx_trace_hunter/config.json`

GUI、命令行默认启动和后台安装脚本会优先使用同一份系统配置；如果安装脚本发现该配置已存在，默认直接复用，不覆盖已有 Webhook。

状态文件和配置文件分开：

- 配置文件保存“想怎么跑”：路径、日期、告警、AI、Webhook 等。
- 状态文件保存“已经扫到哪”：日志路径、偏移、轮询状态、上次告警等。

## 排障建议

如果看起来“卡住”，优先判断是不是以下情况：

- 默认进入驻场模式，本来就不会自动退出。
- 第一次扫描的软件根目录太大，正在发现日志。
- 第一次接管超大日志，正在读取尾部内容。
- 归档文件很多，正在解压和筛选。

排查顺序：

1. 看控制台实时输出。
2. 打开输出目录里的 `hunter.log`。
3. GUI 中点击 `打开日志`。
4. 先用 `--once` 测试单轮分析。
5. 把扫描根目录缩小到具体日志目录。

示例：

```powershell
python nginx_trace_hunter.py D:\logs --once
python nginx_trace_hunter.py D:\logs --once --output-dir D:\trace-output
```

## 性能建议

大日志场景建议：

- 首次测试先用 `--once` 和明确日志目录。
- 历史日志很多时先选择日期范围。
- 驻场模式交给状态文件做增量扫描。
- 不要把扫描根目录指向无关的大盘根目录。
- 只在需要时开启 AI。

脚本会在运行日志中按行数输出进度，例如：

- `全量统计阶段进度 ... lines=10000`
- `异常复核阶段进度 ... lines=10000`

这样可以判断程序是在处理大文件，而不是无响应。

## 常见问题

### Nginx 日志里能看到请求体吗？

默认看不到。标准 Nginx access log 通常只记录 IP、时间、方法、URL、状态码、响应大小、Referer、User-Agent 等。

如果要找 token，优先检查：

- URL 参数。
- Cookie。
- Authorization/Bearer 是否被自定义 log_format 记录。
- WAF 日志。
- 应用日志。
- 网关日志。

### 脚本会不会自动封 IP？

不会。脚本只输出待人工研判 IP 和风险依据，是否封禁由安全人员判断。

### AI 会不会每次都调用？

不会。AI 默认关闭；开启后也只有本地规则发现明显风险时才会做辅助总结。

### 为什么要有状态文件？

驻场模式需要记住已经发现的日志和读取偏移，否则每次启动都可能重复扫描大日志。状态文件让脚本可以重启后继续接管。

## 推荐工作流

应急现场可以按这个顺序：

1. 先用 GUI 或 `--once` 对指定日期范围做历史分析。
2. 查看 `report.md`、`suspicious_ips.csv`、`suspicious_uris.csv`。
3. 对 `review_ips.csv` 中的高优先级 IP 做人工研判。
4. 根据高风险 URI 反查配置、应用目录和代码路径。
5. 配好 Webhook 后启动驻场增量巡检。
6. 后续根据推送中的 IP、URI、文件路径、时间窗口继续人工处置。
